01

Lift & Shift Survival Checklist

Must-haves only - follow these steps in order, every time

⚡ Non-Negotiable Skip any step below and your flow will fail in the target tenant. Do not improvise the order.
01
Pre-Export Checks - Do This Before Touching the Source
Source Tenant · CRG
  • Confirm the flow exists inside a named, non-default Solution. If it lives in the Default Solution, add it to a proper Solution first.
  • Verify all child flows (flows called by this flow) are also inside the same Solution.
  • Check that every connector in the flow has a Connection Reference - not a hardcoded personal connection.
  • Open each Connection Reference and note the connector type (SharePoint, Teams, Dataverse, HTTP, etc.). You will need to recreate these in the target.
  • List all Environment Variables used by the flow. Record their internal names (Schema names) and their current values.
  • If the flow uses any custom connectors, export those separately and note their endpoints and authentication type.
  • Confirm the Solution contains no Flows in a broken or suspended state. Fix or remove them before export.
  • If flows reference SharePoint lists, Dataverse tables, or other data sources - record the exact URLs or table names. These must exist in the target tenant before import.
↓
02
Export as Unmanaged - From CRG Tenant
Source Tenant · CRG
  • Go to make.powerapps.com in the CRG tenant.
  • Navigate to Solutions in the left nav. Select your Solution.
  • Click Export Solution.
  • Select Unmanaged - always use Unmanaged when you need to edit or re-import later. Never use Managed for migration unless the target environment is Production and you intend no further edits there.
  • Click Next, then Export. Save the .zip file to a known location.
  • Keep the zip file name unchanged. Do not rename it.
Managed vs Unmanaged - Quick Rule Use Unmanaged in Dev and Test (you can edit after import). Use Managed in Production (locked down, cannot edit). For this migration, start with Unmanaged in Dev to validate first.
↓
03
Prepare the Target Tenant Before Import
Target Tenant · Thermo Fisher
  • Log in to make.powerapps.com in the Thermo Fisher tenant.
  • Select the correct target environment (Dev, Test, or Prod) from the environment switcher.
  • Create any SharePoint sites, lists, or Dataverse tables that the flow depends on. The import will not create these for you.
  • Create all required Connections in the target tenant. Go to Data → Connections → New Connection and authenticate with the correct service account or credentials for each connector type.
  • If custom connectors exist, import them first before importing the Solution.
  • If the Solution uses Environment Variables, prepare their values. You will set these during or immediately after import.
↓
04
Import Solution Into Target Tenant
Target Tenant · Thermo Fisher
  • In the Thermo Fisher tenant, go to Solutions → Import Solution.
  • Upload the .zip file from your export.
  • On the Import Options screen, you will be prompted to map Connection References. Map each one to the connections you created in Step 03.
  • Do not skip unmapped connection references. An unmapped connection reference means that flow will fail at runtime.
  • Set all Environment Variable values on the import screen. If a field shows the old value from the source, replace it with the target value (e.g., updated SharePoint URL).
  • Click Import and wait for completion. Do not close the browser.
  • On success, verify the Solution appears in the Solutions list with a green status.
⚠ Common Failure If the import shows a warning about Connection References not mapped, stop. Do not proceed. Go back to Step 03 and create the missing connections first.
↓
05
Post-Import: Turn Flows On and Assign Owners
Target Tenant · Thermo Fisher
  • Open the imported Solution. Check each flow - they will likely be turned off by default after import.
  • Open each flow and click Turn on.
  • If a flow is owned by a user who does not exist in the Thermo Fisher tenant, reassign the owner. Go to the flow details and use the "Run only users" or ownership settings.
  • For flows using service accounts, ensure the service account credentials are active in the target tenant and the connections are authenticated.
  • For RPA flows (Desktop flows), confirm the machine connection points to a registered machine or machine group in the target tenant. Old machine connections from CRG will not work.
↓
06
Post-Import Validation - Minimum Required
Target Tenant · Thermo Fisher
  • Open each flow in the editor. Visually inspect that all actions load without red error banners.
  • Trigger the flow manually once using Test → Manually.
  • Check the Run History - confirm the run shows "Succeeded" end-to-end.
  • If any step shows a failure: open that step, check if the connection reference is properly mapped, and verify the target resource (SharePoint list, table, etc.) exists.
  • For Desktop flows: run a test execution from the Power Automate Desktop console on the target machine.
  • Confirm that all child flows also completed successfully during the parent flow run.

Common Failure Points - Know These Before You Start

Failure Why It Happens How to Fix

Flow fails on first action Environment variable still points to source tenant URL Edit Environment Variable value in the Solution after import

02

Explicit Approval to Ignore Best Practices

This section formally grants you permission to move without fixing

✅ Official Stance Move now. Fix later. A messy flow that runs in the target tenant is a success. A perfect flow that never gets migrated is a failure.

✅ You May Safely Ignore These

  • Non-standard naming conventions on flows, actions, or variables
  • Missing error handling or try-catch patterns
  • Inconsistent use of compose vs. initialize variable
  • Redundant or unused actions inside flows
  • Flows with no descriptions or comments
  • Inefficient loops or nested conditions
  • Missing run-only user restrictions
  • Non-standard trigger configurations (as long as they still fire)
  • Overly complex expressions that work but are unreadable
  • Duplicate logic across flows
  • Hardcoded values that are non-connection-related (string labels, etc.)
  • Lack of logging or notification steps

🚫 You Must Not Ignore These

  • Hardcoded tenant-specific URLs (CRG SharePoint URLs inside flow actions - these must be updated)
  • Personal connections used directly - these will not work in the target tenant for another user
  • Missing Connection References - flows using direct connections instead of references will break
  • Flows outside a Solution - cannot be exported properly
  • Child flows missing from Solution - parent flow will fail at the child call step
  • Machine/machine group references tied to CRG - Desktop flows will not run
  • Missing Environment Variables with no default value - flow will error immediately
  • Expired or personal credentials in connections - must be refreshed in target
Reminder The goal of this migration is functional equivalence, not technical excellence. Every quality improvement is a post-migration task. Track issues found during migration in a backlog, not as blockers.
03a

Documentation - Must-Haves

Step-by-step deployment process and migration reference

A
Standard Migration Flow: Source → Target
  • Step 1 - Inventory: List every flow being migrated. Record: flow name, solution name, connectors used, environment variables, child flows.
  • Step 2 - Source preparation: Ensure all components are inside a named Solution in CRG. Fix broken flows or exclude them intentionally.
  • Step 3 - Export: Export Solution as Unmanaged from CRG.
  • Step 4 - Target prep: Create connections, data sources, and environment variable placeholders in Thermo Fisher tenant.
  • Step 5 - Import: Import Solution, map connection references, set environment variable values.
  • Step 6 - Enable: Turn on flows, assign owners, configure machine connections for RPA.
  • Step 7 - Test: Run test cases per the Testing Strategy (Section 03c).
  • Step 8 - Sign-off: Developer confirms all test cases pass. Document any known issues for backlog.
B

C
Handling Environment Variables
  • Environment Variables travel with the Solution export as definitions only. Values are not exported by default (this is by design in Power Platform).
  • During import, the wizard will prompt you to set each variable value. Do not skip this.
  • If you have many variables, prepare a simple text file listing each variable Schema Name and its target value before you start the import.
  • If a variable has a default value set in the definition, that default will be used if you skip setting a value - verify that default is still valid for the target tenant.
  • To update an environment variable after import, go to: Solutions → [Your Solution] → Environment Variables → edit the value.
  • Never hardcode values inside flow actions to work around missing variables. Update the variable instead.
Variable naming reference:
Schema Name:  new_SharePointSiteURL
Source Value:  https://crg.sharepoint.com/sites/MySite
Target Value:  https://thermofisher.sharepoint.com/sites/MySite
D
Managed vs Unmanaged - Decision Rules
EnvironmentSolution TypeReason
Development (Dev) UNMANAGED Developers need to edit flows after import. Unmanaged allows editing.
Test / UAT UNMANAGED Testers may need quick fixes. Use Unmanaged until the solution is stable.
Production MANAGED Managed locks down the solution. No accidental edits. Changes go through proper deployment.
Important Never import a Managed solution into an environment where you have an existing Unmanaged version of the same solution without first removing the Unmanaged one. This causes layer conflicts.
E
Connection Reference Mapping - Step by Step
  • Before import: go to Data → Connections in the Thermo Fisher tenant and create one connection per connector type used in the Solution (SharePoint, Outlook, Teams, Dataverse, etc.).
  • Authenticate each connection with the correct service account - not your personal account - unless the flow is designed for personal use.
  • During import: the wizard shows a list of Connection References. Each reference has a dropdown. Select the matching connection you created.
  • If a connection reference is left as "None", the flow will import but will fail when it tries to use that connector at runtime.
  • After import: you can verify and update connection reference mapping by going to Solutions → [Your Solution] → Connection References.
  • If multiple flows share the same connector type, they should share one Connection Reference - do not create separate references for the same connector.
F
Minimum Validation Checklist - Before Handoff
  • Solution imported without errors (no red status)
  • All flows visible inside the Solution in the target tenant
  • All Connection References mapped to active connections
  • All Environment Variables set to target values
  • All flows turned on
  • Flow ownership assigned to correct account
  • Manual test run completed - status shows "Succeeded"
  • All child flows confirmed executed during parent test run
  • RPA: Desktop flow test completed on target machine / machine group
  • No broken references in flow editor (no red action blocks)
03b

Documentation - Good-to-Haves

Post-migration improvements - do not block migration on these

⏱ OPTIONAL - Post-Migration Only. None of the items below are required for the flow to run. Complete these after successful migration and sign-off.

Code Review Checklist

  • Remove unused actions and dead branches
  • Replace hardcoded strings with Environment Variables
  • Consolidate duplicate logic into child flows
  • Add descriptive names to all actions
  • Document the purpose of each flow with a description field

Naming Conventions

  • Flows: [Area]-[Function]-[Type] e.g. HR-Onboarding-Trigger
  • Variables: camelCase, descriptive
  • Connection References: CR_[ConnectorName]
  • Environment Variables: EV_[Purpose]
  • Solutions: include team name and version

Logging & Error Handling

  • Add a Scope block around critical actions
  • Add a "Configure run after" step set to "has failed"
  • Post failure details to a Teams channel or SharePoint log list
  • Send email alerts on unhandled failures
  • Add timestamps to log entries

Performance Improvements

  • Replace "Apply to each" loops with batch operations where possible
  • Enable concurrency on loops only if order doesn't matter
  • Use "Filter array" before loops instead of filtering inside the loop
  • Reduce unnecessary HTTP calls by caching results in variables
  • Break very large flows into parent/child pattern

Testing Strategy

What to test, where, and how - HMs for Dev, HMGs for Test and Production

A
HMs vs HMGs - When to Use Each
Machine TypeEnvironmentUse ForKey Difference
HM
Hosted Machine		 Development Initial testing, debugging, attended scenarios, fixing connection issues Persistent. Machine stays allocated. Good for interactive debugging.
HMG
Hosted Machine Group		 Test, Validation, Production Unattended execution, scheduled runs, full end-to-end validation Non-persistent. Machines spin up and release. Credentials must be stored securely.
B
What to Test in Development Using HMs
Attended Scenarios
  • Trigger the flow manually and step through each action in the run history.
  • Verify the Desktop flow launches and interacts with the correct application on the HM.
  • Test with real credentials - confirm the service account can authenticate to each target system.
  • Validate that all Environment Variable values resolve correctly at runtime.
  • Run the flow through a complete happy-path scenario - no shortcuts or partial runs.
  • Test error scenarios: wrong input, unavailable resource, timeout. Confirm the flow handles them without hanging.
  • Confirm child flows are triggered and return the expected outputs to the parent.
  • Check the flow run history for any warnings even on a "Succeeded" run.
C
What to Validate in Test and Production Using HMGs
Unattended Scenarios
  • Confirm the flow can be triggered without human interaction (scheduled or event-based trigger).
  • Verify the HMG has at least one available machine at the time of execution - check the machine group capacity in the Power Automate portal.
  • Run the flow via its actual trigger (not manual test) to simulate production conditions.
  • Confirm credentials stored for the HMG are valid - test a login via the machine group directly if possible.
  • Validate that application selectors in the Desktop flow work on the HMG - UI elements sometimes behave differently on non-persistent machines.
  • Check that no step requires interactive input (pop-up dialogs, MFA prompts) - these will cause unattended runs to hang.
  • Confirm run history in Test environment shows consistent results over multiple runs, not just one.
Unattended MFA Warning If any connector or application requires MFA during runtime, the flow will hang and eventually fail in unattended mode on HMGs. Resolve MFA exceptions or use certificate-based auth before moving to Test.

E
How to Validate Connections, Credentials, and Dependencies
  • Connections: Go to Data → Connections. Each connection should show a green status. A red warning icon means the credential is expired or invalid - click it and re-authenticate.
  • Service Account Credentials: Confirm the service account password has not expired. Confirm it is not blocked by Conditional Access policies in the Thermo Fisher tenant.
  • SharePoint / Dataverse: Open the target list or table in a browser with the service account. If you can't read/write manually, the flow won't be able to either.
  • Custom Connectors: Test the connector directly via the test tab in the Custom Connector editor. Confirm the endpoint is reachable from the Thermo Fisher tenant.
  • Machine Connections (RPA): In Power Automate portal, go to Monitor → Machines. Confirm the machine shows as "Available" or "Connected".
  • HTTP Calls: If flows call external APIs, confirm those APIs are accessible from the Thermo Fisher network and any IP allowlisting is updated.

⛔ Failure Handling & Rollback Approach

  • Do not delete the original Solution from CRG until all test cases pass in Thermo Fisher.
  • If import fails mid-way: note the error message, fix the specific issue (missing connection, missing component), and re-import. Power Platform imports are safe to re-run.
  • If flows fail in Test after passing Dev: check if the HMG machine has the required applications installed and if credentials differ from HM credentials.
  • Rollback for cloud flows: revert by disabling the flow in the target and reactivating in the source temporarily.
  • Rollback for RPA flows: redirect the parent cloud flow to call the machine in the source environment by updating the machine connection reference.
  • Keep a backup of the exported .zip file. You can re-import at any point.
  • If a solution is deleted in the target, you can re-import from the zip without re-exporting from source.
